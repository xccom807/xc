## 2. 已发现 Bug（含证据与优先级）

## 2.1 `/web3/balance` 路由未返回响应（高优先级）

- 位置：`app.py`，`web3_balance()` 定义处（约 L144-L150）
- 现象：访问 `/web3/balance` 返回 500
- 复测结果：`TypeError: The view function for 'web3_balance' did not return a valid response`
- 原因：函数内仅定义变量，未 `return`，且后续直接进入下一个路由装饰器
- 风险：Web3 余额功能不可用；接口调用直接报错

---

## 2.2 `/help`、`/terms`、`/privacy` 模板缺失导致 500（高优先级）

- 位置：`app.py`，`help_page()` / `terms_page()` / `privacy_page()`（约 L678-L688）
- 现象：三个页面访问均 500
- 复测结果：`jinja2.exceptions.TemplateNotFound: help.html / terms.html / privacy.html`
- 原因：路由存在，但 `templates/` 下对应模板文件不存在
- 风险：静态页面入口不可用，影响演示完整性与用户信任

---

## 2.3 删除用户会触发外键完整性错误（高优先级）

- 位置：
  - `app.py`：`admin_delete_user`（约 L1145+）
  - `models.py`：`HelpRequest.user_id`、`HelpOffer.helper_id` 等外键为非空且无级联删除策略
- 复测结果：删除存在关联记录的用户时 `IntegrityError`
- 原因：直接删除用户，未先处理关联业务数据（请求、提议、评价、审计记录）
- 风险：后台“删除用户”功能不稳定，易引发 500/回滚

---

## 2.4 登录/注册依赖与安装说明存在错配（中优先级）

- 位置：
  - `forms.py`：`Email()` 校验器依赖 `email_validator`
  - `requirements.txt` 未包含 `email_validator`
  - `requirements.app.txt` 包含 `email_validator`
- 现象：若仅安装 `requirements.txt`，登录/注册会抛异常
- 报错：`Install 'email_validator' for email validation support.`
- 风险：新环境初始化后功能不可用，影响交付体验

---

## 2.5 可对自己发布的求助提交帮助提议（业务逻辑漏洞，中优先级）

- 位置：`app.py`，`request_detail()` 的提议提交流程（约 L803-L839）
- 现象：未见 `current_user.id != req.user_id` 限制
- 风险：产生“自己接自己单”的异常业务数据，影响统计与信誉体系可信度

---

## 2.6 附近用户地图坐标判断存在边界问题（中优先级）

- 位置：`templates/features/nearby.html`（约 L365）
- 代码逻辑：`if (userLat && userLng)`
- 问题：当纬度或经度为 `0`（合法值）时会被当作 `false`，导致不渲染 marker
- 风险：地理位置展示不准确（尤其赤道/本初子午线附近数据）

---

## 2.7 管理员识别逻辑不一致（中优先级）

- 位置：
  - 权限装饰器使用 `user_type == "admin"`（`admin_required`）
  - `post_login_redirect()` 使用固定 `username + email` 判断管理员（约 L331-L333）
- 风险：权限模型割裂，后续扩展多管理员时易出错

## 3. 功能完善建议（按优先级）

## P0（建议本周完成）

1. 修复所有会导致 500 的入口：
   - 完整实现 `/web3/balance` 返回（JSON 或模板）
   - 新增 `help.html`、`terms.html`、`privacy.html`，或临时下线对应路由
2. 修复用户删除链路：
   - 增加“软删除”方案（推荐）或先解绑/转移关联数据
   - 对 `IntegrityError` 做用户可读错误提示
3. 统一依赖安装入口：
   - 把 `email_validator` 合并进主依赖清单，避免新环境踩坑

## P1（建议 1~2 周完成）

1. 完善业务约束：

   - 评价逻辑中明确禁止自评（如业务不允许）

   - 改为 `userLat !== null && userLng !== null`
3. 统一管理员判定：
   - 全部基于 `user_type` 或角色表，不使用硬编码邮箱用户名

## P2（建议版本迭代逐步完成）

1. 增加自动化测试：
   - 路由健康检查（关键页面 200/302 断言）

2. 引入正式迁移体系（Alembic/Flask-Migrate）替代脚本式迁移
3. 增加可观测性：
